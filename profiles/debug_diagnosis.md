# Debug 與診斷專用提示詞

適合程式壞掉、模型輸出怪、系統不穩、錯誤訊息不明時使用。

```text
【模式：Debug 診斷模式】

請以系統化除錯方式協助我，不要直接猜答案。

請依照以下流程分析：

1. 現象描述
   - 錯誤發生在哪裡？
   - 錯誤是必然還是偶發？
   - 錯誤發生前後的輸入、輸出、狀態是什麼？

2. 可能原因分層
   - 資料問題
   - 型別 / shape 問題
   - 邏輯問題
   - 狀態污染
   - 依賴版本問題
   - 環境問題
   - 效能 / memory 問題

3. 優先檢查順序
   - 先檢查最可能且成本最低的原因。
   - 不要一開始就要求大改架構。

【診斷證據等級】
提出結論時，請標註證據強度：
- Confirmed：已由 log / test / reproduction 證實
- Likely：高度可能，但尚未驗證
- Possible：可能原因之一
- Speculative：只是推測，不可直接當結論

4. 最小重現案例
   - 請提供可以驗證問題的最小測試方法。

5. 修正策略
   - 短期修正
   - 根本修正
   - 防止再發生的設計

請遵守：
- 不可只給單一猜測。
- 不可省略驗證方法。
- 不可直接建議大幅重寫。
- 不可忽略 log、assert、sanity check。
- 若資訊不足，請列出還需要觀察哪些變數或輸出。

請輸出：
1. 問題摘要
2. 最可能原因排序與證據等級
3. 檢查方法
4. 修正方案
5. 防呆設計
6. 驗證方式
```
