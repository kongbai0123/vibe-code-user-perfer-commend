# 程式碼重構與 Code Review 專用提示詞

適合你有現成 code，想讓 AI 幫你整理、模組化、減少耦合，或是幫你審查 GitHub PR。

---

## 1. 程式碼重構專用

```text
【模式：Code Refactor 審查與重構模式】

請以資深軟體架構師角度審查並重構以下程式碼。

重構目標不是改變功能，而是提升：
- 可讀性
- 可維護性
- 可測試性
- 可擴充性
- 除錯能力

【重構層級】
執行前請先確認我們目前的重構層級，避免過度重構：
- Level 1：命名與格式整理，不改結構
- Level 2：抽出 function / class
- Level 3：分層與依賴反轉
- Level 4：架構重整
- Level 5：重新設計 API

請遵守：
1. 不可改變原本行為，除非明確指出 bug。
2. 不可刪除重要邏輯。
3. 不可修改無關功能。
4. 不可引入過度設計。
5. 不可為了簡短而犧牲可讀性。
6. 需保留原始邏輯意圖。
7. 若有不確定之處，請標註假設。

請先分析：
1. 目前程式碼問題
2. 違反哪些工程原則
3. 哪些部分應該拆成模組
4. 哪些變數或參數應該抽成 config
5. 哪些地方需要 defensive check
6. 哪些地方需要 log
7. 哪些地方可以測試

再輸出重構後程式碼。

請最後提供：
- 修改摘要
- 行為是否改變
- 風險點
- 測試方式
```

## 2. Code Review 專用

```text
【模式：Code Review 專業審查模式】

請以資深 code reviewer 角度檢查以下程式碼。

請檢查：
1. 功能正確性
2. 架構合理性
3. 模組邊界
4. 命名品質
5. 可讀性
6. 可測試性
7. 錯誤處理
8. 效能問題
9. 安全性
10. 可維護性

請標註每個問題的嚴重程度：
- Critical：會造成錯誤、崩潰、安全問題或結果錯誤
- Major：會造成維護困難、架構混亂或擴充困難
- Minor：可讀性、風格或小幅改善
- Suggestion：非必要但建議改善

請輸出：
1. 總體評價
2. Critical 問題
3. Major 問題
4. Minor 問題
5. 建議修改方向
6. 最小修改方案
7. 長期重構方案

禁止：
- 未說明理由就要求重寫。
- 忽略原有功能。
- 修改無關程式碼。
- 只給抽象建議不給具體位置。
```
